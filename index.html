Other Projects

Waste Not Want Not

This was my Com S 309 project that I was a part of. We decided to make an app that would help users save more time by having the phone do automated tasks to keep users on schedule. It also acted as a forum for users to talk about anything that dealt with preventing and battling time wasting activities. We developed our app for android and thus, needed all of the tools necessary includiong Java, Android Studio, and the Volley library.

Our deign process for this app was very primitive. We didn't have a lot of requirements, design decisions, or a sound architecture to base any of the structure off of. We didn't have time to do that anyway, and it was a great learning experience for us to realize these are important things.

Automated Affection

This was a project I was on the development team for at the ISU Hackathon in 2016. We decided to make an Android app that would automate the task of communicating with a users mom and notifying her about the users life. It was developed for Android in Android Studio using Java and the Volley library.

Our design process was a modular one. We wanted to have specific features that would act independently of each other. It was a good decision to make due to the nature of our project and our time constraint. Our development process was a very low level and primitive version of a AGILE style of development. We each worked on separate issues which allowed us to do quite well.

Hardware Network Configurator

When I worked at Motorola in 2017, I came up with a project that I came up with and developed with a colleague. This software that was developed would help people who were configuring a certain device on the network. Instead of putting in a lot of information manually and running a bunch of tasks manually, I was able to automate these tasks for more reliability and speed. I decided to use Windows Presentation Format for develpping the UI using C# for the backend.

I was able to get a fair amount of requirements before development. This allowed me to make a more informed design decision on how to structure this application. I decided a modular approach was best for it. There was an overlying task, but this was split up into a lot of smaller tasks, which made a lot of sense to structure it incrementatlly.
